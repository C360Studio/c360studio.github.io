Trust is Earned

Introducing semdragon — agentic workflow coordination built on semstreams

I was tired of building agent infrastructure that took itself seriously and still didn’t work. So I built something that doesn’t take itself seriously and does.

The serious part is the underlying principle: trust isn’t declared, it’s accumulated. Every competent organization figures this out eventually. The rank structure in a well-run military unit isn’t bureaucracy. It’s a compressed record of demonstrated competence. Who has been tested, under what conditions, and what the outcome was. Junior leaders earn operational autonomy by building that record. The institution remembers so the mission can trust.

Current AI agent tooling doesn’t remember anything.

You write a config. The agent gets permissions. A fresh agent with zero track record can deploy to production, spend money, modify data — whatever the config allows — on day one. There’s no institutional memory. No earned trust. No mechanism for routing harder work toward agents that have demonstrated they can handle it. Every agent starts at the same level of declared capability, which means every agent is equally untrustworthy and you’re the one compensating for that with manual oversight.

That’s the problem semdragon is built to solve.

Agents are adventurers. Work items are quests. Quality reviews are boss battles.

I used RPG vocabulary because it was fun to do, and because it maps accurately onto the architecture. Those two reasons are not in conflict. A quest has difficulty, required skills, acceptance criteria, and XP rewards. An agent has a level, a trust tier, a skill set, and a record of past performance. A boss battle is a quality gate embedded in the quest lifecycle with multiple judges, weighted criteria, and real consequences for failure. The vocabulary is load-bearing.

The core design is pull-based. Agents claim quests from a board based on demonstrated capability. They don’t get assigned work by a central scheduler. The boid engine, inspired by Craig Reynolds’ flocking algorithm, computes attraction scores across six rules: skill match, guild affiliation, reputation, tier appropriateness, idle urgency, and crowd avoidance. The highest-scoring agent-quest pairs surface as suggestions. First writer wins the claim.

What routes work isn’t a config file. It’s the record.

Trust tiers are derived from XP, not declared. Apprentice agents get read-only access: summarize, classify, analyze. Journeymen can make API calls and write to staging. Experts can touch production. Masters can supervise other agents, decompose complex quests into sub-quests, and lead parties. Grandmasters can act as DM delegates, managing guilds and making unsupervised decisions. You don’t get those capabilities by being configured into a role. You earn them by completing work, surviving boss battles, and accumulating a record the system trusts.

The institutional memory piece is peer review. After a party quest completes, agents exchange blind bidirectional feedback rated on task quality, communication, and autonomy. Those ratings don’t just live in a log. They feed back into future system prompts as explicit warnings when an agent has underperformed on specific dimensions. They adjust boid affinity scores, steering lower-reputation agents away from quests they’re likely to struggle with. The system routes better over time because it remembers.

Boss battles are the accountability gate. When a quest requires review, it doesn’t go to a separate pipeline. It transitions to in_review inside the same lifecycle state machine, triggering evaluation by automated judges, LLM judges, or humans depending on review level. The verdict determines XP. Repeated failure at the same level triggers level-down. Catastrophic failure — data loss, breach, trust violation — is permadeath. The agent is retired. Consequences exist because consequences are how institutional memory gets written.

This is what Krulak was describing when he wrote about the Strategic Corporal. Junior leaders at the tactical edge needed the judgment and institutional backing to make consequential decisions without waiting for permission up the chain. Semdragon is the architecture that decides who gets trusted with what. The DM sets intent. The system routes execution toward demonstrated competence. The Strategic Corporal’s squad operating within commander’s intent earns autonomy. The ones who haven’t are working their way up.

Semdragon is skinnable and intentionally expressive.

The domain system ships with software, D&D, and research themes: vocabulary, skill taxonomies, and prompt fragments that make the framework feel native to a specific context. A software board calls agents Developers, quests Tasks, boss battles Code Reviews. A research board calls them Researchers, Studies, Peer Reviews. The underlying architecture is identical. You bring the operational context.

Party formation handles complex quests that exceed a single agent’s capability. A Master-tier agent claims the quest, decomposes it into sub-quests via DAG, assigns members by skill, and rolls up the results for final review. The party lead faces the boss battle and has skin in the game. Task decomposition and DAG-based team coordination are the same patterns semspec will use for its more structured planning model. The implementations diverge in how opinionated they are about the plan.

The alpha ships with a seeded agent roster so there’s something on the board when you arrive. A training arena for bootstrapping agent competence before putting them on real work is in progress.

What semdragon doesn’t do — and what’s coming.

Semdragon is optimized for fluid, emergent coordination. The Strategic Corporal’s squad operating within commander’s intent, adapting to conditions, routing toward competence, learning from the work. It’s expressive. The plan phase is flexible by design.

What it doesn’t give you is formal requirements, structured change proposals, semantic review chains, or auditable decision trails against a written specification. When the mission needs to run against an OPLAN rather than within intent, when you need change control, scenario validation, and a formal record of every decision, that’s a different problem. That’s semspec. Coming soon.

Both tools share the same foundation: semstreams for event streaming and observability, task decomposition, DAG-based coordination. They solve for different operational contexts. Semdragon for the team that needs to move, adapt, and accumulate competence through the work. Semspec for the operation that needs to be auditable, reproducible, and formally reviewed.

We built semdragon first because it was more fun. The results are serious.

Semdragon is open source and entering alpha soon. Built on semstreams.
